---

## Guiding Principle

Every ticket must deliver **verifiable behavior**: something you can confirm works via store E2E tests, unit tests that exercise real logic, or manual UI testing. No ticket exists solely to "define types" or "set up architecture." If a ticket introduces new types, it also uses them in working code within the same ticket.

The one exception is **large refactoring** (like the boolean→number migration) where the verification is zero regression across the full test suite.

---

## Ticket Naming Convention

`P05-NN` where NN is a zero-padded sequence number.
Each ticket targets ONE coherent feature with its own tests and commit(s).

### P05-02: Multi-bit data model migration (boolean → number)

**Phase section:** 0.5.2
**Task ID:** T-BUS-MODEL
**Effort:** 8h

**Addresses gap:**
- **GAP-3D-2** (Multi-bit Buses — foundational): the gap states "All values are `boolean` (not multi-bit integers)" and "The simulation engine only propagates `boolean` values." This ticket changes the data model from `boolean` to `number` across the entire codebase, unblocking all subsequent bus work (P05-11, P05-12, P05-13).

**Scope:** Change `Pin.value`, `InputNode.value`, `OutputNode.value` from `boolean` to `number`. This is a refactoring ticket — verified by zero regression across the full test suite + build.

**Files:**
- Modify: `src/store/types.ts` — value types
- Modify: `src/simulation/gateLogic.ts` + test — number inputs/outputs
- Modify: `src/store/actions/simulationActions/simulationActions.ts` + test
- Modify: `src/store/actions/wireActions/*.ts` + `nodeActions/*.ts`
- Modify: all 3D components reading `.value`
- Modify: E2E tests referencing boolean values

**Strategy:**
1. Change types, let TypeScript errors guide every call site
2. Gate logic: wrap boolean functions — `(inputs: number[]) => number`
3. `getSignalSourceValue` returns `number`
4. 3D components: `value === 1` as "on"
5. Node toggle produces `0`/`1`

**Verification:** `pnpm run lint && pnpm run test:run && pnpm run test:e2e:store && pnpm run build` all green. No behavioral change — existing circuits work identically.

**Exit:** All existing tests pass with `number` values. Build succeeds.

### P05-03: Topological sort simulation

**Phase section:** 0.5.6
**Task ID:** T-TOPO
**Effort:** 10h

**Addresses gap:**
- **GAP-3D-5** (Simulation Engine Limitations — full resolution): the gap documents that the 3-step single-pass tick requires N ticks for N-layer circuits, producing wrong `eval` results and visible glitches. This ticket implements topological-sort evaluation so a single `eval` produces correct, fully-propagated results — matching the web-ide's approach.

**Scope:** Replace the 3-step simulation tick with topological-sort single-pass evaluation. After this ticket, multi-layer circuits evaluate correctly in a single `eval` — the core correctness fix.

**Files:**
- Create: `src/simulation/topologicalEval.ts`
- Create: `src/simulation/topologicalEval.test.ts`
- Modify: `src/store/actions/simulationActions/simulationActions.ts` — `simulationTick` calls `topologicalEval`

**What to build:**
- `topologicalSort(gates, wires, inputNodes, outputNodes)` → evaluation order
- `evaluateCircuit(state: CircuitState)` → single-pass, correct results
- Cycle detection: combinatorial feedback → error
- Plug into existing `simulationTick`

**Verification:**
- **New tests:** Two-layer (NOT→AND), three-layer (NOT→AND→OR), fan-out — all correct in single eval
- **Regression:** All existing simulation + E2E store tests pass
- **Manual:** Open app, place NOT feeding AND, toggle input — output updates correctly in one step (not two)
- Cycle detection: A→B→A → error reported

**Exit:** `pnpm run test:run && pnpm run test:e2e:store` green. Multi-layer circuits work.
