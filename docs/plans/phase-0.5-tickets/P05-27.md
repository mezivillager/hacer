## Context

All preceding tickets (P05-01 through P05-26) deliver individual features. This ticket integrates them into full student workflows and proves they work end-to-end. The three primary workflows:

### 1. HDL Path (most common for nand2tetris)
```
Select chip → load HDL stub → edit HDL → auto-compile → run .tst → see results
```

### 2. 3D Path (HACER's unique value)
```
Select chip → auto-generate I/O → place gates → wire → validate → run .tst → see results
```

### 3. Mixed Path (advanced)
```
Build NOT in 3D → export → register as chip
Build AND in HDL using that NOT → compile → run And.tst → passes
```

### Supporting flows tested:
- **Persistence**: Save → reload → load → verify state
- **Builtin toggle**: Switch to builtin → all 16 tests pass
- **Curriculum navigation**: Browse chips, track completion

## Pitfalls

- **Dependency chain**: HDL path tests for And/Or/Mux require compiling dependency chips first (Not → And → Or). Each test must set up the full chain or share a fixture.
- **Window globals**: Store E2E tests operate via `window.__CIRCUIT_ACTIONS__` and `window.__CIRCUIT_STORE__`. New actions (compileHDL, runChipTest) must be exposed there.
- **Test isolation**: Each test should start with a clean store + clean registry. Avoid leaking chips between tests.
- **3D path complexity**: The 3D path for And requires fan-out (one signal to two destinations). This needs junction support. If junction E2E helpers don't exist, build them.
- **Performance**: Running all 16 builtin tests sequentially should complete in <5s total. If slow, check for unnecessary re-renders.
- **Flakiness**: Store-level E2E tests should be deterministic (no timing dependencies). Avoid `waitForTimeout` — use store state assertions.
