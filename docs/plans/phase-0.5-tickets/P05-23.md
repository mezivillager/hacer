## Context

P05-15 registered all 16 Project 1 builtins in the `ChipRegistry` via `registerProject1Builtins()`. P05-01 provides the registry API. This ticket adds a UI toggle so the student can:
1. Switch between their own implementation and the reference builtin
2. Experiment with the builtin to understand a chip's expected behavior before implementing it
3. Use builtins as fallback chip-parts when the student hasn't built a dependency yet

### Nand2tetris recommendation:
> "Before implementing a chip, it is recommended to experiment with its builtin implementation."

### Evaluation mode state:

```typescript
// In store:
evaluationMode: 'student' | 'builtin'  // default: 'student'
```

When `'builtin'`, the test engine and hierarchy evaluator resolve chips from builtins only. When `'student'`, they resolve from the student's compiled/circuit chips first, falling back to builtins for chips the student hasn't implemented yet.

### `src/core/chips/resolverFactory.ts`

Factory that creates a `ChipResolver` (P05-17 interface) based on evaluation mode:

```typescript
import type { ChipRegistry } from './registry'
import type { ChipResolver, ChipEvaluator } from '@/core/testing/nand2tetris/engine'
import { evaluateChip } from './evaluate'

export function createChipResolver(
  builtinRegistry: ChipRegistry,
  studentRegistry: ChipRegistry,
  mode: 'student' | 'builtin'
): ChipResolver {
  return {
    resolve(chipName: string): ChipEvaluator | null {
      // In builtin mode, only use builtins
      if (mode === 'builtin') {
        return createEvaluatorFromDef(builtinRegistry.get(chipName))
      }

      // In student mode: student first, then fallback to builtin
      const studentChip = studentRegistry.get(chipName)
      if (studentChip) return createEvaluatorFromDef(studentChip)
      return createEvaluatorFromDef(builtinRegistry.get(chipName))
    },
  }
}

function createEvaluatorFromDef(chipDef: import('./types').ChipDefinition | undefined): ChipEvaluator | null {
  if (!chipDef) return null
  const inputs: Record<string, number> = {}
  const outputs: Record<string, number> = {}

  for (const pin of chipDef.inputs) inputs[pin.name] = 0
  for (const pin of chipDef.outputs) outputs[pin.name] = 0

  return {
    setInput(name, value) { inputs[name] = value },
    evaluate() {
      const result = evaluateChip(chipDef, { ...inputs }, 0)
      Object.assign(outputs, result)
    },
    getOutput(name) { return outputs[name] },
    getInput(name) { return inputs[name] ?? 0 },
  }
}
```

### `src/core/chips/resolverFactory.test.ts`

```typescript
import { describe, it, expect, beforeEach } from 'vitest'
import { createChipRegistry, registerBuiltin } from './registry'
import { createChipResolver } from './resolverFactory'
import type { ChipRegistry } from './registry'

let builtinReg: ChipRegistry
let studentReg: ChipRegistry

beforeEach(() => {
  builtinReg = createChipRegistry()
  registerBuiltin(builtinReg, 'Not',
    [{ name: 'in', width: 1 }], [{ name: 'out', width: 1 }],
    (i) => ({ out: i.in === 0 ? 1 : 0 })
  )

  studentReg = createChipRegistry()
})

describe('createChipResolver', () => {
  it('builtin mode: resolves from builtins', () => {
    const resolver = createChipResolver(builtinReg, studentReg, 'builtin')
    const not = resolver.resolve('Not')!
    not.setInput('in', 0)
    not.evaluate()
    expect(not.getOutput('out')).toBe(1)
  })

  it('student mode: falls back to builtin when student chip missing', () => {
    const resolver = createChipResolver(builtinReg, studentReg, 'student')
    const not = resolver.resolve('Not')!
    not.setInput('in', 1)
    not.evaluate()
    expect(not.getOutput('out')).toBe(0)
  })

  it('student mode: prefers student chip over builtin', () => {
    registerBuiltin(studentReg, 'Not',
      [{ name: 'in', width: 1 }], [{ name: 'out', width: 1 }],
      () => ({ out: 42 }) // obviously wrong — but proves student is preferred
    )
    const resolver = createChipResolver(builtinReg, studentReg, 'student')
    const not = resolver.resolve('Not')!
    not.setInput('in', 0)
    not.evaluate()
    expect(not.getOutput('out')).toBe(42)
  })

  it('returns null for unknown chip', () => {
    const resolver = createChipResolver(builtinReg, studentReg, 'builtin')
    expect(resolver.resolve('Unknown')).toBeNull()
  })
})
```

## Pitfalls

- **Two registries**: Keep builtins and student chips in **separate** registries (or partition one). The resolver factory picks from the right one based on mode.
- **`evaluateChip` depth**: Nested HDL must thread `depth + 1` per P05-18; `createEvaluatorFromDef` should call `evaluateChip(chipDef, { ...inputs }, 0)` only at the **top-level** resolver — inner compiled builtins already increment depth if P05-16 contract (A) is followed.
- **`getOutput` / P05-17**: Return `undefined` for unknown output pin names (use `outputs[name]` without defaulting to `0`) so the test engine can fall back to `getInput` for columns that are inputs.
- **Fallback is per-chip**: In student mode, the resolver falls back chip-by-chip. If the student has built And but not Not, And is student's, Not is builtin. This is essential for incremental work.
- **Store state is simple**: `evaluationMode` is just `'student' | 'builtin'`. No closures, fully serializable.
- **Antd `Switch` mock**: In tests, mock as a `<button>` with `onClick` that calls `onChange(!checked)`.
- **React Compiler**: No `useMemo`, `useCallback`, `React.memo`.
