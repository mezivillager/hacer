## Context

Every value in HACER is currently `boolean`: `Pin.value`, `InputNode.value`, `OutputNode.value`, gate logic functions, wire propagation, and 3D component rendering all use `true`/`false`.

Nand2tetris Project 1 requires 16-bit buses. The data model must change from `boolean` to `number` (integer bitmask) so that a single pin can carry 0–65535.

This is a **refactoring ticket** — no new features. Verified by zero regression.

---

## Goal

Change `value: boolean` to `value: number` everywhere. After this ticket:
- Single-bit values use `0` and `1` (not `true`/`false`)
- Gate logic accepts `number[]` and returns `number`
- Wire propagation carries `number`
- 3D components display `value === 1` as "on" (visually identical)
- All existing tests pass with `number` values

## Implementation Strategy

1. **Start with types**: Change `src/store/types.ts` — `Pin.value`, `InputNode.value`, `OutputNode.value` from `boolean` to `number`.
2. **Run `pnpm run typecheck`** — TypeScript will show every broken call site. This is your migration guide.
3. **Fix gate logic**: Change `gateLogic.ts` functions. The key transformation:
   ```typescript
   // Before:
   export const nandGate = (a: boolean, b: boolean): boolean => !(a && b)
   // After:
   export const nandGate = (a: number, b: number): number => (a & b) === 0 ? 1 : 0
   ```
   For all gates, convert boolean operators to bitwise on single bits:
   - `!(a && b)` → `(a & b) === 0 ? 1 : 0` (NAND)
   - `a && b` → `a & b` (AND)
   - `a || b` → `(a | b) !== 0 ? 1 : 0` (OR — for single-bit: `a | b`)
   - `!a` → `a === 0 ? 1 : 0` (NOT)
   - `a !== b` → `a ^ b` (XOR — for single-bit only)
   - `a === b` → `a ^ b ^ 1` (XNOR — for single-bit only)

   The `gateLogic` record type becomes `Record<GateType, (inputs: number[]) => number>`.

4. **Fix simulation actions**: `getSignalSourceValue` returns `number`. Replace `?? false` with `?? 0`.
5. **Fix node/wire actions**: Default values `false` → `0`. Toggle `!value` → `value === 0 ? 1 : 0`.
6. **Fix 3D components**: Change `value ? '1' : '0'` to `String(value)` or `value === 1 ? '1' : '0'`. Color: `value === 1 ? onColor : offColor`.
7. **Fix test files**: Bulk replace `true` → `1` and `false` → `0` in value contexts.
8. **Fix E2E**: Same boolean-to-number replacement in assertions and scenarios.

After each group, run `pnpm run typecheck` to track remaining errors.

## Gate Logic Conversion Reference

```typescript
// src/simulation/gateLogic.ts — AFTER migration
import type { GateType } from '@/store/types'

export const nandGate = (a: number, b: number): number => (a & b) ^ 1
export const andGate = (a: number, b: number): number => a & b
export const orGate = (a: number, b: number): number => a | b
export const notGate = (a: number): number => a ^ 1
export const norGate = (a: number, b: number): number => (a | b) ^ 1
export const xorGate = (a: number, b: number): number => a ^ b
export const xnorGate = (a: number, b: number): number => (a ^ b) ^ 1

export const gateLogic: Record<GateType, (inputs: number[]) => number> = {
  NAND: (inputs) => nandGate(inputs[0], inputs[1]),
  AND: (inputs) => andGate(inputs[0], inputs[1]),
  OR: (inputs) => orGate(inputs[0], inputs[1]),
  NOT: (inputs) => notGate(inputs[0]),
  NOR: (inputs) => norGate(inputs[0], inputs[1]),
  XOR: (inputs) => xorGate(inputs[0], inputs[1]),
  XNOR: (inputs) => xnorGate(inputs[0], inputs[1]),
}
```

These bitwise operations only work correctly for single-bit (0 or 1) values. Multi-bit bus simulation (P05-11) will add proper multi-bit propagation later.

## Pitfalls

- **Do NOT change behavior** — this is a type migration only. `0`/`1` must behave identically to `false`/`true` in every existing test.
- **JavaScript truthiness**: `if (value)` still works because `0` is falsy and `1` is truthy. But explicit comparisons like `value === true` will break. Search for these.
- **E2E tests run in the browser** — `page.evaluate()` returns actual values, so boolean comparisons like `=== true` must become `=== 1`.
- **`setInputValue` in wireActions** — the param is `boolean` today. Change to `number` and update callers.
- **Use `pnpm run typecheck` iteratively** — do not try to fix everything in one pass. Fix types → check → fix next batch.
- **Do NOT add new features** — no multi-bit simulation, no bus ops, no width changes. Those come in P05-11.
