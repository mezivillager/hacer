## Context

After P05-16 (HDL compiler) and P05-14 (persistence / serialize), HACER has two separate build modes:
- **3D mode**: Wire gates on canvas → `SerializedCircuit` → evaluate via `topologicalEval`
- **HDL mode**: Type HDL text → `parseHDL` → `compileHDL` → `ChipDefinition` → evaluate via `evaluateChip`

These modes don't talk to each other yet. A chip built in 3D cannot be used as a part in HDL, and a chip compiled from HDL cannot be placed on the 3D canvas (the latter is partially addressed by P05-24 for rendering, but the wiring-to-evaluation loop needs this ticket).

### What interop means:

1. **3D → HDL**: Export a 3D circuit to HDL source. This requires auto-generating internal wire names for connections that are not chip-level I/O.
2. **HDL → 3D**: A compiled HDL chip is already a `ChipDefinition` in the registry. P05-24 renders it. This ticket ensures the evaluation path works end-to-end.
3. **Mixed chains**: NOT built in 3D, then AND built in HDL using that NOT as a part.

### Internal wire naming:

In a 3D circuit, wires connect pin-to-pin directly. When exporting to HDL, each wire that connects a gate output to another gate's input (not a chip I/O) needs a name. Strategy:

```
Gate "nand1" output "out" → wire → Gate "not1" input "in"
HDL: Nand(a=a, b=b, out=w1);   // w1 is auto-generated
     Not(in=w1, out=out);
```

The `Wire.signalId` field (currently optional/unused) can store these names.

After this ticket:
1. `exportCircuitToHDL(state, chipName)` → HDL source string with correct CHIP declaration, pin declarations, PARTS section with auto-named internal wires
2. **`importHDLToCircuit` (optional in this ticket)**: Full reverse compilation (HDL → placed gates + wires) is large; **defer** to a follow-up unless you already have placement heuristics. Minimum for P05-26: **export** + **registerCircuitAsChip** + **evaluate `type: 'circuit'`**.

### `src/core/chips/interop.ts`

```typescript
import type { ChipDefinition, ChipPin } from './types'
import type { CircuitState, GateInstance, Wire, GateType } from '@/store/types'

/** Map canvas `GateType` to HDL PARTS chip name (course uses `Nand`, not `NAND`). */
function mapPrimitiveGateTypeToHDLPart(type: GateType | string): string {
  const table: Record<string, string> = {
    NAND: 'Nand', AND: 'And', OR: 'Or', NOT: 'Not', XOR: 'Xor', NOR: 'Nor', XNOR: 'Xnor',
  }
  return table[type] ?? type
}

function cloneCircuitState(state: CircuitState): CircuitState {
  return structuredClone(state)
}

export interface HDLExportResult {
  hdl: string
  wireNames: Map<string, string>  // wireId → generated name
}

export function exportCircuitToHDL(
  state: CircuitState,
  chipName: string,
  chipInputs: ChipPin[],
  chipOutputs: ChipPin[]
): HDLExportResult {
  const wireNames = new Map<string, string>()

  // Internal signal names: assign one HDL identifier per **logical net** (fan-out). Do **not** use the old
  // `!isFromChipInput && !isToChipOutput` heuristic — it misclassifies wires.
  // Practical v1: require `wire.signalId` on internal nets; v2: BFS/union-find from gate outputs through junctions.
  for (const wire of state.wires) {
    if (wire.signalId) wireNames.set(wire.id, wire.signalId)
  }
  // After graph analysis, ensure every gate-to-gate segment shares the same `wireNames` value as its driver output.

  // Build PARTS section
  const parts: string[] = []
  for (const gate of state.gates) {
    // Map `GateType` / composite `chipName` → HDL PARTS name (`NAND` → `Nand`, `COMPOSITE`+chipName → `Not`, …)
    const chipPartName = gate.chipName ?? mapPrimitiveGateTypeToHDLPart(gate.type)
    const connections: string[] = []

    // Map input pins
    for (const input of gate.inputs) {
      const wire = state.wires.find(w => w.to.entityId === gate.id && w.to.pinId === input.id)
      if (wire) {
        const signalName = resolveSignalName(wire, 'from', state, wireNames)
        connections.push(`${input.name}=${signalName}`)
      }
    }

    // Map output pins
    for (const output of gate.outputs) {
      const wire = state.wires.find(w => w.from.entityId === gate.id && w.from.pinId === output.id)
      if (wire) {
        const signalName = resolveSignalName(wire, 'to', state, wireNames)
        connections.push(`${output.name}=${signalName}`)
      }
    }

    parts.push(`    ${chipPartName}(${connections.join(', ')});`)
  }

  const inDecl = chipInputs.map(p => p.width > 1 ? `${p.name}[${p.width}]` : p.name).join(', ')
  const outDecl = chipOutputs.map(p => p.width > 1 ? `${p.name}[${p.width}]` : p.name).join(', ')

  const hdl = [
    `CHIP ${chipName} {`,
    `    IN ${inDecl};`,
    `    OUT ${outDecl};`,
    `    PARTS:`,
    ...parts,
    `}`,
  ].join('\n')

  return { hdl, wireNames }
}

function resolveSignalName(
  wire: Wire,
  direction: 'from' | 'to',
  state: CircuitState,
  wireNames: Map<string, string>
): string {
  const endpoint = direction === 'from' ? wire.from : wire.to
  if (endpoint.type === 'input') {
    const node = state.inputNodes.find(n => n.id === endpoint.entityId)
    return node?.name ?? 'unknown'
  }
  if (endpoint.type === 'output') {
    const node = state.outputNodes.find(n => n.id === endpoint.entityId)
    return node?.name ?? 'unknown'
  }
  return wireNames.get(wire.id) ?? 'unknown'
}

export function registerCircuitAsChip(
  name: string,
  inputs: ChipPin[],
  outputs: ChipPin[],
  state: CircuitState
): ChipDefinition {
  return {
    name,
    inputs,
    outputs,
    implementation: { type: 'circuit', circuitData: cloneCircuitState(state) },
  }
}
```

## Pitfalls

- **Pin name resolution**: The export function must map wire endpoints to the correct signal names. An endpoint connected to an InputNode maps to the node's name. An endpoint connected to another gate maps to an auto-generated internal wire name.
- **P05-18 coordination**: If `evaluate.ts` still throws for `type: 'circuit'`, implement that path **here** (or in the same PR as this ticket) so `registerCircuitAsChip` is testable.
- **`structuredClone`**: Use for immutable snapshots; if targeting a constrained environment, replace with the serialization round-trip from P05-14.
- **Fan-out**: One gate output may connect to multiple destinations (via junctions). All wires in that fan-out group share the same signal name.
- **Circuit evaluation performance**: Evaluating a circuit-type chip means running `topologicalEval` on a snapshot. For small circuits this is fine; for deep nesting, consider caching the evaluation function.
- **Serialization**: `circuitData` in `ChipImplementation` holds a `CircuitState` snapshot. This is JSON-serializable (unlike closures). P05-14's serialize/deserialize can handle it.
- **HDL round-trip fidelity**: Export → re-import may not produce identical circuit state (gate positions, wire routing differ). The semantic behavior must match: same inputs → same outputs.
- **`Wire.signalId` adoption**: If a wire already has a `signalId`, use it as the HDL wire name. Otherwise auto-generate. This lets users pre-name wires for cleaner HDL output.
