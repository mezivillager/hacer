## Context

HACER's `simulationTick` in `src/store/actions/simulationActions/simulationActions.ts` uses a 3-step single-pass evaluation:

1. For each wire → if dest is gate input, copy source value (reads gate outputs from **previous** tick)
2. For each gate → evaluate logic, write new output values
3. For each wire → if dest is output node, copy source value

This means a circuit with N gate layers requires N ticks to fully propagate. A NOT→AND chain (2 layers) gives wrong results after a single `eval`. The nand2tetris test scripts call `eval` once and expect the correct answer immediately.

The simulation currently operates inside a Zustand+Immer `set()` call. `getSignalSourceValue()` resolves values by tracing wire sources (input nodes, gate outputs, junctions with cycle detection).

## Algorithm

### Topological Sort

Build a dependency graph where each gate depends on its input sources:

1. **Nodes in the graph**: Each gate is a node. Input nodes are source nodes (no dependencies). Output nodes are sink nodes.
2. **Edges**: For each wire, if the wire goes from gate A's output to gate B's input, gate B depends on gate A.
3. **Kahn's algorithm**: Start with gates that only depend on input nodes (in-degree = 0). Process them, remove their outgoing edges, add newly-zero-in-degree gates to the queue.
4. **If the queue empties before all gates are processed**: There's a cycle — report it.

### Single-pass evaluation


## Pitfalls

- **Immer / Zustand tests**: Outside of `set((state) => { ... })`, `useCircuitStore.getState()` is **not** an Immer draft. In Vitest, prefer `useCircuitStore.setState((state) => { evaluateCircuit(state) })` or pass a **plain** `CircuitState` clone into `evaluateCircuit` if you test the pure function in isolation.
- **Immer compatibility**: `evaluateCircuit` receives a draft state from Immer. Mutations like `gate.outputs[0].value = result` work on drafts. Do not spread/clone — mutate in place.
- **Junction handling**: Junctions are wire fan-out points, not gates. They should be transparent in the topological sort — trace through them to find the actual gate or input node source.
- **Wire endpoints**: Wires use `WireEndpoint` with `type: 'gate' | 'input' | 'output' | 'junction'` and `entityId`/`pinId`. Build the dependency graph from these.
- **Gates have multiple input pins**: A NAND has `inputs[0]` and `inputs[1]`. Each may be fed by a different wire from a different source. The gate depends on ALL of its input sources.
- **Empty circuits**: Handle gracefully — no gates, just input/output nodes directly wired.
- **Keep `getSignalSourceValue` working**: Some E2E tests or other code may call `simulationTick` and check intermediate state. The behavior should be the same (or better — correct in one tick instead of needing N).
