## Context

After P05-02 (boolean→number), the data model supports multi-bit values. Students building 16-bit chips like Not16 need visual tools:
- **Bus splitter**: takes a 16-bit bus and exposes 16 individual 1-bit outputs
- **Bus joiner**: takes N individual 1-bit inputs and produces an N-bit bus output
- **Thick wires**: visually distinguish multi-bit buses from 1-bit wires

### Current wire rendering:

Wires are rendered in `src/components/canvas/` as thin lines with `WireSegment[]` data. There is no width-based visual differentiation.

### Current gate components:

All gates in `src/gates/components/` share a similar structure: a `<group>` with a mesh body, pin connectors, and text labels. Bus components follow the same pattern but with dynamic pin counts.

---

## Goal

After this ticket:
1. `placeBusSplitter(width, position)` creates a splitter node: 1 input (N-bit) → N outputs (1-bit each)
2. `placeBusJoiner(width, position)` creates a joiner node: N inputs (1-bit each) → 1 output (N-bit)
3. Splitter and joiner render as 3D components on the canvas with labeled pins
4. Wires with `width > 1` render thicker with a `×N` label
5. Splitter/joiner are wirable like other components

## Files to Create

### `src/store/actions/busActions/busActions.ts`

```typescript
import type { Position, GateInstance, Pin } from '@/store/types'

function createPins(count: number, type: 'input' | 'output', prefix: string): Pin[] {
  return Array.from({ length: count }, (_, i) => ({
    id: `${prefix}${i}`,
    name: `${prefix}${i}`,
    type,
    value: 0 as any, // number after P05-02
  }))
}

export interface BusActions {
  placeBusSplitter: (width: number, position: Position) => GateInstance
  placeBusJoiner: (width: number, position: Position) => GateInstance
}

export const createBusActions = (set: any, get: any): BusActions => ({
  placeBusSplitter: (width: number, position: Position): GateInstance => {
    // Creates a special gate-like entity with:
    // - 1 input pin named 'in' (width bits)
    // - `width` output pins named 'out0'..'outN'
    // Implementation: add to gates array with a special type marker
    // or add to a new busComponents array in state
    let result!: GateInstance
    set((state: any) => {
      const id = `splitter-${Date.now()}-${Math.random().toString(36).slice(2, 8)}`
      const instance: GateInstance = {
        id,
        type: 'BUS_SPLITTER' as any,
        position,
        rotation: { x: 0, y: 0, z: 0 },
        inputs: [{ id: 'in', name: 'in', type: 'input', value: 0 as any }],
        outputs: createPins(width, 'output', 'out'),
        selected: false,
      }
      state.gates.push(instance)
      result = instance
    }, false, 'placeBusSplitter')
    return result
  },

  placeBusJoiner: (width: number, position: Position): GateInstance => {
    let result!: GateInstance
    set((state: any) => {
      const id = `joiner-${Date.now()}-${Math.random().toString(36).slice(2, 8)}`
      const instance: GateInstance = {
        id,
        type: 'BUS_JOINER' as any,
        position,
        rotation: { x: 0, y: 0, z: 0 },
        inputs: createPins(width, 'input', 'in'),
        outputs: [{ id: 'out', name: 'out', type: 'output', value: 0 as any }],
        selected: false,
      }
      state.gates.push(instance)
      result = instance
    }, false, 'placeBusJoiner')
    return result
  },
})
```

**Design note:** Bus components reuse the `GateInstance` type with a new `type` discriminator (`'BUS_SPLITTER'` / `'BUS_JOINER'`). This requires extending `GateType` or using a broader discriminated union. The simplest approach for Layer 1: extend `GateType` to include these types and add them to gate logic as passthrough operations. Alternative: add a separate `busComponents` array — discuss trade-offs at implementation time and pick whichever avoids the most churn.

### `src/gates/components/BusSplitter3D.tsx`

```tsx
import { Text } from '@react-three/drei'
import type { GateInstance } from '@/store/types'

interface BusSplitter3DProps {
  gate: GateInstance
  onPointerDown?: (e: any) => void
}

export function BusSplitter3D({ gate, onPointerDown }: BusSplitter3DProps) {
  const width = gate.outputs.length
  const bodyHeight = Math.max(1.2, width * 0.15)

  return (
    <group position={[gate.position.x, gate.position.y, gate.position.z]} onPointerDown={onPointerDown}>
      {/* Trapezoidal body: wide on output side */}
      <mesh>
        <boxGeometry args={[1.2, 0.6, bodyHeight]} />
        <meshStandardMaterial color={gate.selected ? '#ffcc00' : '#4a9eff'} />
      </mesh>
      {/* Label */}
      <Text position={[0, 0.35, 0]} fontSize={0.18} color="#fff" anchorX="center">
        SPLIT ×{width}
      </Text>
      {/* Input pin (left side) */}
      <mesh position={[-0.7, 0, 0]}>
        <sphereGeometry args={[0.08]} />
        <meshStandardMaterial color="#00ff88" />
      </mesh>
      {/* Output pins (right side, spaced vertically) */}
      {gate.outputs.map((_, i) => (
        <mesh key={i} position={[0.7, 0, (i - (width - 1) / 2) * 0.2]}>
          <sphereGeometry args={[0.06]} />
          <meshStandardMaterial color="#ff8800" />
        </mesh>
      ))}
    </group>
  )
}
```

## Files to Modify

### Wire rendering (location TBD — `src/components/canvas/` area)

When rendering a wire, check `wire.width` (default 1):
- `width === 1`: render as current thin line
- `width > 1`: render thicker (e.g. `lineWidth * 2`) and add a `<Text>` label showing `×N` near the wire midpoint

### `src/store/types.ts`

Either extend `GateType` to include `'BUS_SPLITTER' | 'BUS_JOINER'`, or introduce a broader `ComponentType` union. The former is simpler but conceptually mixes primitives with bus utilities; the latter is cleaner but requires more refactoring. Choose at implementation time.

### `src/simulation/gateLogic.ts`

Add logic entries for splitter (copy input bits to individual outputs) and joiner (combine input bits into output). These are simple passthrough operations using `readSubBus` / `writeSubBus` from P05-11.

### Canvas rendering (gate renderer)

Add a branch in the gate rendering logic that dispatches to `BusSplitter3D` or `BusJoiner3D` based on the gate type.

## Pitfalls

- **`GateType` union**: Adding new strings to the union type will cause TypeScript exhaustiveness checks to fail everywhere `GateType` is switched on (gate selector, gate logic, 3D rendering). Fix each switch/map. Alternatively, use a separate entity type to avoid this churn.
- **Pin positions in 3D**: Splitter/joiner have many pins — calculate positions dynamically based on count. Ensure `getPinWorldPosition` works for these components (may need to extend `pinPositionConfig`).
- **Wiring compatibility**: The existing wiring system (`startWiring`, `completeWiring`) works with gate pins by ID. Splitter/joiner pins follow the same pattern — no wiring changes needed if the pin IDs are unique and `getPinWorldPosition` returns correct positions.
- **React Compiler**: No `useMemo` for pin position arrays. Just compute inline.
- **Don't implement sub-bus wiring UI** — that's an HDL compiler concern (P05-16). This ticket is about the visual components and their store representation.
